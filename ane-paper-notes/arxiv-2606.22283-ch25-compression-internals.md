# 第 25 章 压缩内部机制（Compression Internals）通俗解读

> 原论文：Spencer H. Bryngelson，《Apple Neural Engine: Architecture, Programming, and Performance》，arXiv:2606.22283，2026 年 6 月。
> 本章原文位于论文第 158–164 页。本文为基于原文的通俗中文解读，不是逐字翻译。

---

## 一、这一章到底在讲什么？

**一句话**：讲 Apple Neural Engine（ANE）这个矩阵加速器，**怎么把神经网络里几千万、上亿个权重（weight）压扁了存、再在芯片里现场还原成可计算的数**。

为什么这件事很重要？

- 神经网络里 99% 的数据量都是权重。
- 权重从内存搬到芯片的"带宽"，决定了推理快慢和耗电。
- 全精度（FP16）搬太慢、太费电，所以必须压缩。
- ANE 在芯片里**内置了几套硬连线解压电路**，让权重以压缩形态流过来、边流边解、立刻送进乘加阵列。

这一章就是把这五套压缩格式 + 两道"是否走流式解压"的开关 + 一个"权重打包记录"格式 + 几个边界情况，从头到尾拆开讲清楚。

## 二、先建立全景图：权重的"一生"

```
  训练好的 FP32 权重
        │
        │  Core ML 转换器（量化 / 调色板化 / 稀疏化）
        ▼
  压缩字节流 + 一份描述符（descriptor）
        │
        │  编译器决定：流式解压？还是先展开成 FP16？
        ▼
  ┌─────────────┐    要么  ┌──────────────┐
  │ DMA 直接搬运 │        │ 先在 host 展开 │
  │ 压缩字节流   │        │ 成 FP16 再搬  │
  └─────────────┘        └──────────────┘
        │                        │
        ▼                        ▼
   芯片内硬连线解压         （已经解压完了）
        │                        │
        ▼                        ▼
          送进 MAC 乘加阵列计算
```

整章围绕三件事展开：

1. **有哪些压缩格式？**（25.1 + 25.2）
2. **怎么决定走哪条路？**（25.3）
3. **打包记录长什么样？怎么被芯片用起来？**（25.4）

剩下的 25.5–25.8 是四个专题：稀疏、FP8、版本差异、Winograd。

## 三、格式枚举：每种权重都有自己的"身份证号"

### 3.1 一个 1~31 的整数标识格式

每一个编译好的权重 blob 开头都带一个 `format enum`（格式枚举），是 1 到 31 之间的一个整数。这个数字就是它的"身份证号"。

有 **5 张并行查找表** 都以这个数字为 key，分别告诉你：

| 帮助函数 | 返回什么 |
|---|---|
| `ZinKernelFormatGetBitDepth(f)` | 每个元素占几位 |
| `ZinKernelFormatGetPaletteFormat(f)` | 调色板子格式（仅对 7~27 有效） |
| `ZinKernelFormatGetUnderlyingType(f)` | 底层数据类型（int8 / uint8 / e4m3 / FP16） |
| `ZinKernelGetPaletteLUTSize(f, n)` | 码本字节大小 × 码本个数 |
| `ZinKernelFormatGetTypeno(f)` | 写进描述符的类型编号 |

> 类比：身份证号一报出来，公安系统立刻告诉你这个人的血型、身高、户籍。这里也一样：format enum 一给，五张表马上告诉你这个权重怎么读。

### 3.2 两个关键的"区间"

- **调色板格式**：enum 落在 **7 ~ 27**（用判断 `f - 7 < 0x15` 拦出来）。
- **Unity 格式**（恒等/只缩放，跳过乘法阵列的特殊权重）：enum 落在 **28 ~ 30**（判断 `f - 0x1c < 3`）。

这两个区间后面会反复出现。

### 3.3 一份权重 = 六个固定段

编译好的权重单元由固定的六段组成，由描述符（descriptor）一一列出：

```
t%u          → 元信息
lut:         → 激活函数查找表
pal:         → 调色板查找表
ocgs:        → 输出通道分组（OCG）表 ★后面会重点讲
scale:       → 每输出通道的缩放系数
bias:        → 每输出通道的偏置
weights:     → 真正打包好的权重或索引数据
```

注意：**激活 LUT 和调色板 LUT 是平级的**，共享同一块片上内存预算——你占得多，我就得少占。

## 四、四种压缩编码（25.2 的核心）

这是本章最关键的一张表。把每种格式的"反量化公式 + 长什么样 + 哪代芯片能流式 + 谁来开关"都列出来了。

### 4.1 总览表

| 格式 | 反量化关系 | 在线形态 | 哪代芯片原生流式 | 编译器开关 |
|---|---|---|---|---|
| **int8 仿射**（per-tensor / per-channel） | `w = s·(q − z)`，M1 上 `z = 0` | int8 字节流 + 每通道 FP16 scale | A14+ | M1 上**折叠**到 dense |
| **int4 查找表**（LUT / 调色板） | `w = LUT[g/v][k][c mod v]` | 4-bit 索引流 + 16 项 FP16 码本 | M1/H13+ | 流式主开关 + kernel-stride enable |
| **结构化稀疏** | 用 mask 把非零值散播回去 | 1-bit mask + 压缩 FP16 非零值 | M1/H13+ | 流式主开关，作为独立操作数 |
| **块级仿射**（blockwise affine） | `w = s_b·q` 每块一个 scale | int8 字节流 + 每块 FP16 scale | A15+ | M1/M2 上**折叠**到 dense |

下面挨个说人话。

### 4.2 ① int8 仿射量化（最经典）

存的就是 8 位整数 `q`，用时这样还原：

```
w = s · (q − z)
```

- `q`：存的 8 位整数。
- `s`：scale，FP16 浮点。可以一个张量共用一个，也可以**每输出通道一个**。
- `z`：零点（zero point）。

**M1 上的限制**：强制 `z = 0`，公式退化成 `w = s·q`（对称量化）。非对称设置在 M1 上会直接硬断言失败。

> 为什么每通道 scale 不增加额外计算？
> 因为编译器在**编译期**就把 scale 和 bias 折进了权重系数，存进了描述符的 scale/bias 数组里。运行时引擎在 gain-offset 阶段直接套用：
>
> ```
> y = s_out · Σ(w'_k · x_k) + b_out
> ```
>
> 也就是说：**对称量化卷积在 M1 上不为反量化付一分钱运行时开销**。

### 4.3 ② int4 查找表 / 调色板（M1 上唯一原生流式的格式）

存的是 4 位**索引** `k`，指向一张 16 项的 FP16 码本（codebook）：

```
w = LUT[g/v][k][c mod v]
```

- `g`：输出通道组（OCG）编号。
- `v`：调色板向量大小（连续多少个输出通道共用一张码本）。
- `k`：4 位索引。
- `c`：通道位置。

**为什么 4-bit 只能走调色板？** 因为类型表里压根没有 4-bit 的"标量类型"——4 位数据只能是索引，不可能是数值本身。

**per-tensor 的常见情况**：`v = 1`，整张表共用 1 个 16 项码本，公式退化成一次表查找。

支持的索引位宽：2/4/6/8 bit。但 M1 实际只跑 **4-bit（16 项）** 和 **8-bit（256 项）**：
- 1/2-bit：通用路径拒绝。
- 3/6-bit：被版本门控到后续表描述符。
- 调色板+量化组合：码本最多 256 项。

**一个有趣的合法性 mask**：向量大小 v 只能取 `{1, 2, 4, 5}`。
判断方法：把 `2^v` 跟 `0x36 = 0b110110` 比，命中的就合法。也就是说 v = 0、3、≥6 都不行。

**大小**：4-bit 码本 32 字节，8-bit 码本 512 字节。

### 4.4 ③ 结构化稀疏（structured sparsity）

不是把权重数值变小，而是**把 0 直接不存**。

存两块东西：
```
mask     : ceil(n/8) 字节      每个元素 1 bit，LSB 优先
nonzeros : 2·popcount(mask) 字节   保留的 FP16 非零值，按扫描顺序
```

还原（伪代码）：

```python
def densify_sparse(mask_bits, nonzeros, n):
    out = [0.0] * n
    j = 0
    for i in range(n):
        if mask_bits[i]:
            out[i] = nonzeros[j]
            j += 1
    return out
```

只要权重一半以上是 0，就比 dense 划算。

**M1 实测**：一个 6× 卷积栈、零占 63% 的权重，压缩后是 dense 的 0.43 倍字节，跑得快 **1.55~1.64 倍**，等效带宽从 ~29 GB/s 涨到 ~48 GB/s（这就是 M1 的权重流上限）。

### 4.5 ④ 块级仿射（blockwise affine）

跟 int8 仿射几乎一样，但**不是每通道一个 scale，而是每一小块（block）一个 scale**。粒度更细，量化误差更低。

在 M1/M2 上**没有原生流式编码**，会走 dequantize-to-dense 路径，先在 host 端展开成 FP16 再搬。

## 五、流式 vs 折叠：两道开关的拉锯（25.3）

这是本章第二个核心点。压缩了的权重到底怎么送到芯片？有两条路：

- **流式（stream）**：压缩字节直接 DMA 搬到芯片，**片上硬连线**现场解压。
- **折叠（fold）**：在 dispatch 之前，在 host 端先展开成 FP16 dense 张量，再以全宽度搬。

谁走哪条路由**两道协作开关**决定：

### 5.1 主开关：kernel-streaming bit @ `0x48f`

这是**总闸**。只有满足以下全部条件，才允许流式：

1. 硬件抽象层（HAL）的 streaming master 位（偏移 `0x48f`）= 1。**从 A13 起开始置位**。
2. 格式是 primary 或可由 DMA 转 FP16。
3. 所有轴 stride = 1（unit stride）。
4. 无 dilation、无 tile 重叠。
5. 权重**不可变**（mutable 的、就地训练的参数不能流式）。

### 5.2 副开关：palette 路径 @ `0x520`–`0x539`

专门给调色板路径用的。int4 LUT 只有满足：

- 权重是**向量调色板化**的；
- per-format 的 kernel-stride-enable 位（偏移 `0x529`）= 1；

编译器才会真正打开 palette-enable 和 stream 标志。

**M1 实测结果**：

| 格式 | 0x48f | 0x529 | 结果 |
|---|---|---|---|
| int4 LUT | ✓ | ✓ | **流式** |
| 结构化稀疏 | ✓ | — | **流式**（走主开关，作为独立操作数） |
| int8 仿射 | ✓ | ✗（0x520–0x539 清零） | **折叠** |
| 块级仿射 | ✓ | ✗ | **折叠** |

### 5.3 带宽预算的"放松"

走流式的权重按**放宽的**片上内存上限（`0x210`）算；
走折叠或 dense 的按**dense 上限**（`0x200`）算，M1 上是 **64 KB**。

→ 这意味着压缩路径每层能塞下更多权重。

## 六、权重打包：每输出通道一组（OCG）记录（25.4）

权重段是一串 **OCG 记录（per-output-channel-group record）**。每条记录是一个**固定 14 字**的块：

| 字号 | 字段 | 初始值 |
|---|---|---|
| `[0]` | 该 OCG 在段内的字节偏移 | 编译期算出 |
| `[1]` | 该 OCG 的通道数 | 编译期算出 |
| `[2]`–`[0xb]` | **10 个地址重定位槽** | `-1` |
| `[0xc]`, `[0xd]` | 保留标志位 | `0` |

那 10 个 `-1` 槽是干什么的？**等加载器（loader）来"现场填地址"**。填进去的是：

- 输入/输出 tile 的基地址
- 每 buffer 的系数子 buffer 基地址
- **4 个系数流基地址**：bias、post-scale、palette LUT、activation LUT

这 4 个流跟芯片里 **4 个 kernel DMA 子通道**一一对应：

| 描述符槽 | Enable 寄存器 | Base-offset 寄存器 | Relocation 寄存器 |
|---|---|---|---|
| bias | `0x5548` | `0x554c` | `0x1554` |
| post-scale | `0x5558` | `0x555c` | `0x1558` |
| palette LUT | `0x5568` | `0x556c` | `0x155c` |
| activation LUT | `0x5578` | `0x557c` | `0x1560` |

> 没有专门的 zero-point 流，因为 M1 是对称量化，零点恒为 0。

### 6.1 M1 特别之处：每核心一份副本

M1 把 kernel 系数**每个引擎核心复制一份**（4 核 M1 就有 4 份独立的 OCG 向量），而 A14+ 改用共享 kernel 内存布局。

### 6.2 激活查找表（activation LUT）的结构

激活 LUT 和调色板 LUT 共享同一份预算。激活 LUT 是一个 **43 项 FP16** 记录：

- 2 个 input-clamp 值
- 1 个 mode 标志
- 1 个 scale
- **33 个 knot 值**（在输入域里均匀分布）
- 4 个尾部外推系数
- 1 个打包 mode 字

运行时：把输入映射到 33 个 knot 之间的 32 段，整数部分选段，小数部分做 FP16 线性插值，输出区外按端点 knot 钳位。

## 七、稀疏数据通路（25.5）

引擎里其实有**两套独立的稀疏机制**，别搞混：

### 7.1 ① 计算时跳零（zero-skip）

- 表描述符里有个 `detect-zeros` 位（M1 已实现）。
- 打开后，**乘加阵列遇到权重为 0 就跳过这次乘加**。
- 代价模型扫一遍权重算出零密度，缓存起来，按比例削减卷积周期估算。
- **与存储格式无关**——任何含 0 的 kernel 都吃这个红利，包括那些折叠成 dense 存的。

**但有个重要局限**：在一个**带宽受限**的层里，zero-skip 只能把极限推高约 **1%**。因为瓶颈在"权重字节流"而不是"乘加阵列"，跳过乘加帮不上忙。

### 7.2 ② 稀疏二进制存储 + 片上解压（真正省带宽的那个）

- mask 和压缩 nonzeros 走 DMA 流过来，**片上电路**解压成 dense tile。
- 内核配置寄存器里有一个 packed sparse-format 字段，跟 palette-enable、palette-bits 共用一个字。
- 这是 25.4 提到的"作为独立操作数走主开关 0x48f"的那条路径。
- **效果**（M1 实测）：6× 卷积栈、63% 零密度，sparse 流是 dense 的 **0.43 倍字节**，速度 **1.55–1.64×**。
- 编译出的程序字节**完全一致**（2416 字节）——差别只在权重 payload 和描述符。这是**原生流式**的指纹，不是折叠，也不是程序变了。

### 7.3 一个看似矛盾的 assert

version-five 表描述符**断言 sparse-binary 模式不支持**。但实际 sparse 又能流。为什么？

因为那个 assert 管的是**另一条**路径——调色板索引平面内部的 sparse 打包（只有走 vector-palette 标志才能到达）。我们说的能流式的 sparse，是 25.4 那条独立的 mask+values 操作数路径，**不经过那个表描述符位**。

## 八、FP8 数据通路（25.6）——M1 看得到用不了

编译器里有完整的 FP8 通路，但**没有任何 M1 代芯片能跑**。FP8 其实是两个东西：

| 形式 | 元素代码 | 是什么 |
|---|---|---|
| **E4M3** | `0xc` | gated 硬件权重 + 激活格式 |
| **E5M2** | `0xd` | 转换格式 |

E4M3 才是被门控的那个：

- 内核格式合法性**承认**它是原生权重元素类型；
- 但写它元素代码的 encoder，**A17 起才存在**；
- 运行时能力字节 `0x52d` **只在 A18 上才置位**；
- M1 上格式寄存器只有 2 位宽，**根本没有 E4M3 的码点**——直接触发**编译期 assert**（不是运行时拒绝）。

**累加器永远是宽的 FP16-class**——FP8 只是输入/存储宽度，喂给同一组乘加阵列。吞吐走的是跟 int8 同样的"双乘"路径。

完整的 FP8 通路、格式寄存器差异、E5M2↔FP16 折叠不对称、转换函数 → 第 36 章细讲。

## 九、版本矩阵：M1 vs 后续芯片（25.7）

同一份 kernel 格式，**不同表描述符版本会编出不同的权重字节**。所以 M1 和 M5 是**比特级不同**，不只是标量极限不同。

| 能力 | M1 version-5 描述符 | 后续描述符 |
|---|---|---|
| sparse-binary 索引打包 | **assert 不支持** | 描述符 `0x424` 一个 packed flag |
| 多码本调色板 | **assert 不支持** | 已实现 |
| 非对称量化 | **assert 不支持** | 已实现 |
| zero-skip 检测 | 已实现 | 已实现 |
| 调色板使能 + 位宽 | 4/8 bit | 4/8 + 3/6 bit |

**M1 合法 kernel-format 空间**（与 version-5 描述符的交集）：

- dense FP16
- 4-bit / 8-bit 调色板（含单码本向量形式）
- 对称 int8（折叠到 FP16）
- Unity 格式

**M1 完全没有**：sparse-binary 索引打包、多码本调色板、非对称量化、FP8、3-bit/6-bit 调色板宽度。注意：这是**缺 encoder**（编译期就没有），不是运行时拒绝。

## 十、Winograd 与压缩权重的交互（25.8）

Winograd 是一种把小卷积（典型 3×3）变快的算法。完整资格门控在第 20 章推导。

**与压缩相关的核心点**：资格要求权重**非 unity、非 sparse**。

- sparse 或 unity 格式的权重 → **排除在 Winograd 路径之外**，走自己的数据通路。
- 变换矩阵 `G`、`B⊤`、`A⊤` **常驻在阵列里**，不存任何可发货权重 blob → 没有任何 codec 持有它们。

## 十二、读完这一章你应该能回答的问题
