## 1. HAL 表在哪里

HAL 表是**编译器二进制文件里的一份常驻芯片档案**——不在磁盘配置文件、不在内核、不在固件，而是**嵌在编译器可执行文件本身**。

```
┌───────────────────────────────────────────────────────────────┐
│                  ANE 编译器二进制（一份）                         │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│   代码段（一份）：                                              │
│     ┌───────────────────────────────────────────────────┐     │
│     │  共享的编译逻辑（全部代际共用）                       │     │
│     └───────────────────────────────────────────────────┘     │
│                                                               │
│   数据段（每代一份）：                                          │
│     ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌─────────┐ │
│     │ HAL@A11    │ │ HAL@A13    │ │ HAL@M1     │ │ HAL@M5  │ │
│     │ 0x938 字节  │ │ 0x938 字节  │ │ 0x938 字节  │ │ ...     │ │
│     └────────────┘ └────────────┘ └────────────┘ └─────────┘ │
│         ↑                                                       │
│   C 结构体名：ZinIrHalParameters                                │
│   总长：0x938 字节（固定）                                        │
│                                                               │
└───────────────────────────────────────────────────────────────┘

         编译时按 target chip 选对应的表，传给编译器函数
```

**关键事实**：

| 属性 | 值 |
|---|---|
| 物理位置 | 编译器二进制的数据段 |
| C 结构体名 | `ZinIrHalParameters` |
| 总长度 | `0x938` 字节（固定，每代一样大） |
| 每代一份 | A11/A12/.../A18、M1/.../M5 各自有一张 |
| 读取方式 | **内联 `ldrb`/`ldr` + 固定偏移**（**没有 getter 方法**） |
| 字段名来源 | 只活在编译器保留的 mangled C++ 符号里（如 `f(ZinIrHalParameters const&)`） |

**一份编译器代码 + N 张数据表 = 全代际支持**。芯片代际差别**只是数据，不是代码**。

## 2. 什么时候使用

HAL 表**只在编译时被读取**，运行时引擎不读它。论文里反复出现的使用时机可以归纳成四类：

```
┌──────────────────────────────────────────────────────────────────┐
│                    HAL 表的 4 个使用时机                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ①  算子合法性判定（两道闸门同时通过才能原生发射）                    │
│     ┌───────────────────────────────────────────────┐            │
│     │ 闸门 A：MinimumFamily<N>                       │            │
│     │   target_family >= op.minimum_family ?          │            │
│     │   例：sin floor=A15，M1=family2 → 分解；M5=family6 → 原生 │  │
│     ├───────────────────────────────────────────────┤            │
│     │ 闸门 B：hal[offset] & 1                         │            │
│     │   控制算子内部路由（如纹理引擎 0x81d 是否启用）   │            │
│     └───────────────────────────────────────────────┘            │
│                                                                  │
│  ②  数值上限检查                                                  │
│     张量宽度、kernel 维度、工作集大小等                              │
│     超 HAL 表里的值 → 编译拒绝或拆分                                │
│                                                                  │
│  ③  Kernel 内存双轨制（最实用的副作用）                              │
│     if (is_streamable && hal[0x48f]): cap = 16 MB   # 流式权重    │
│     else:                       cap = 64 KB    # 稠密权重         │
│                                                                  │
│  ④  算子内部路径选择                                                │
│     例如纹理引擎在/不在决定 resize 走硬件还是软件分解                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**两道闸门的真值表**（最容易踩坑的地方）：

| 闸门 A（`MinimumFamily`） | 闸门 B（`hal[offset]`） | 结果 |
|---|---|---|
| 家族够格 | 能力字节 = 1 | **算子原生发射** |
| 家族不够格 | — | 分解为更基础的合法算子 |
| 家族够格 | 能力字节 = 0 | 走软件分解路径 |
| — | — | 都不达标 → 编译拒绝 |

> **注意**：HAL 表里有"1"**不等于**真能跑。3D 卷积是经典反例：HAL 表 `0x70` 写着核深度=16、前端识别、但**后端 lowering 在所有设备上都失败**。即 **"attested is not reachable"（登记在册 ≠ 能跑）**。唯一可靠的判定方式是 **compile-and-run on the target**。

## 3. HAL 表里有什么

`0x938` 字节分成两大区域：**标量字段区**（数值上限）+ **能力字节区**（布尔开关）。

### 3.1 区域总览

| 区域 | 偏移范围 | 内容 | 读取方式 |
|---|---|---|---|
| **标量字段区** | `0x18` – `0x348` | 数值上限/限制 | `value = hal[offset]` |
| **能力字节区** | `0x48f` – `0x8cc` | 约 **165 个单字节布尔开关** | `if (hal[offset] & 1) { ... }` |

### 3.2 标量字段区（numeric limits，精选）

| 偏移 | 字段名 | 含义 | M1 的值 |
|---|---|---|---|
| `0x1b8` | `max_operand_bytes` | 片上 SRAM 工作集上限 | 2 MB |
| `0x1c0` | `dram_alignment` | DMA 对齐粒度 | 16 字节 |
| `0x238` | `num_nes` | NE 核数量 | 4 |
| `0x138` | `max_tensor_width` | 张量最大宽度 | 16384 |
| `0x70` | `max_large_conv_kernel_dim_z` | 3D 卷积核深度上限 | 16 |
| `0x200` | dense kernel-memory cap | **非流式**权重上限 | **64 KB** |
| `0x210` | streamed kernel-memory cap | **流式**权重上限 | **16 MB** |

> `0x200` vs `0x210` 是最实用的两个字段——它们决定了"单层权重超过多少会被切碎"。

### 3.3 能力字节区（capability bytes，精选 24 个已破译）

约 165 个单字节布尔开关里，目前 24 个已被命名。下面是最关键的子集：

| 偏移 | 含义 | M1 | A14+ | 备注 |
|---|---|:-:|:-:|---|
| `0x48f` | **流式权重主控** | 1 | 1 | 切换 64 KB ↔ 16 MB kernel 上限 |
| `0x81d` | **纹理引擎** | **0** | **1** | M1 上 resize/crop-resize/affine 走软件分解 |
| `0x52d` | fp8 E4M3 格式 | 0 | 仅 A18 | — |
| `0x4a9` | dropout / 随机 | 0 | A15+ | — |
| `0x815` | 原生 softmax | 1 | 1 | — |

> **M1 上 `0x81d` = 0** 是最常踩的坑：resize、crop-resize、affine transform 全部得走软件分解路径。从 A14 起才有硬件纹理引擎。

### 3.4 算子按 `MinimumFamily` 分层速查

| 等级 | 适用代 | 代表算子 |
|---|---|---|
| **F0** | 全部 | 卷积、矩阵乘、池化、elementwise、reshape、transpose、concat |
| **F2 (A13+)** | A13 起（含 M1） | softmax、各种 norm、reductions、attention、erf、sqrt |
| **F3 (A14+)** | A14 起 | crop-resize、resample |
| **F4 (A15+)** | A15 起 | sin、cos、argmin/argmax |

> **重点**：**没有哪个计算算子的 floor 高于 A15**。A16/A17/A18 这些新代主要增加的是**核数和频率**，而不是新算子。

### 3.5 各代标量参数横向对比

| 字段 | M1 (A13) | A14 | A15 | A16 (M4) | A17 (M5) |
|---|---|---|---|---|---|
| `num_nes` 核数 | 4 | 4 | 4 | 4 | 16（Pro 版） |
| SRAM 工作集 | 2 MB | 2 MB | 2 MB | 2 MB | 2 MB |
| 张量最大宽 | 16384 | 16384 | 16384 | 65536 | 65536 |
| L2 驻留阈值 | 0 | 0 | 32768 | 262144 | 262144 |
| 指令对齐 | 256 | 16 | 16 | 16 | 16 |
| 支持的图像格式数 | 3 | 13 | 16 | 14 | 14 |

核数按 die 等级递增：base = 4、`g` = 8、`s` = 16、`c` = 32、`d` = 64。

## 4. 三个值得记住的判断

1. **HAL 表是"一份编译器 + N 张数据表"模式的核心**。芯片代际差别是数据，不是代码。

2. **两道闸门同时通过才能原生发射**：`MinimumFamily<N>` 管"算子级别"，`hal[offset] & 1` 管"算子内部路由"。M1 上的纹理相关 op 是闸门 B 卡住的典型。

3. **"Attested is not reachable"**——HAL 表里写着支持**不等于**真能跑。3D 卷积是经典反例。任何"原生支持"声明都必须以 **compile-and-run on the target** 为准。
