## 0. 为什么单独拆出来

阶段 5 是"权重的一生"中**最关键的岔路**——同一个权重 blob 走哪条路，决定了它**到底省不省带宽、能不能塞进单层、要不要被切碎**。

把决策树和两条路径拆开看，会发现：
- **决策树**是一组**串联的硬条件**，每道都必须过
- **流式路径**有自己的物理流程（压缩字节 → 片上硬连线解压）
- **折叠路径**有完全不同的物理流程（host 端展开 → dense FP16 → 直通）
- 两条路径**走不同的 kernel-memory 上限**、**不同的 DMA 子通道配置**、**不同的程序描述符位**

**关键观察**：

- **Q1–Q5 是所有格式共享的硬条件**——任何一道失败就折叠
- **Q6–Q7 是 palette 格式（int4 LUT）的额外副开关**——只有 enum 7~27 才查 `0x529`
- **结构化稀疏走 Q1–Q5，不经过 Q6–Q7**——它是独立操作数，不进 palette 路径
- **int8 / 块级仿射走 Q1–Q5，也不经过 Q6–Q7**——但它们被另一条隐含门挡住：M1 上 `0x520`–`0x539` 整簇清零，意味着 per-format stride enable 不开，于是 Q6 失败 → 折叠

## 2. 流式路径（Stream）的完整流程

```
┌──────────────────────────────────────────────────────────────────┐
│                     流式路径（Stream）                            │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  起点：权重 blob（压缩 payload）                                  │
│  · int4 LUT:    4-bit 索引 + 16 项 FP16 码本                     │
│  · 结构化稀疏:  1-bit mask + FP16 nonzeros                       │
│  · int8 (A14+): int8 字节 + 每通道 FP16 scale                    │
│  · 块级 (A15+): int8 字节 + 每 block FP16 scale                  │
│                                                                  │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  S1: kernel-memory 上限 = 16 MB (HAL[0x210])                     │
│  · 单层能塞下大得多 的权重                                         │
│  · 避免被 split-legalize 切碎                                     │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  S2: OCG 记录保持压缩形态                                         │
│  · weights 段是压缩字节，不是 dense FP16                          │
│  · palette LUT 段是 4-bit/8-bit 码本                              │
│  · 4 个系数流基地址填进 OCG relocation 槽                          │
│    （bias / post-scale / palette LUT / activation LUT）           │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼  每次 dispatch
┌──────────────────────────────────────────────────────────────────┐
│  S3: 压缩字节通过 kernel DMA 子通道搬到芯片                        │
│  · 字节数 = 压缩后大小（不是 dense FP16 大小）                     │
│  · 带宽占用小                                                    │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  S4: 片上硬连线解压（在乘法器入口处）                              │
│                                                                  │
│  int4 LUT:      k → codebook[k]           （纯查表，无算术）       │
│  结构化稀疏:    mask + nonzeros → scatter  （回 dense tile）       │
│  int8 (A14+):   q → s·q 或 s·(q-z)                                │
│  块级 (A15+):   q → s_block·q                                      │
│                                                                  │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  S5: 还原成 FP16，送进 MAC 阵列                                   │
│  · 走和 dense FP16 完全相同的乘加 + 宽累加器                       │
│  · 累加精度无损                                                  │
└──────────────────────────────────────────────────────────────────┘

★ 性能指纹：
  · 编译出的程序字节跟"用 dense FP16 编译"的程序字节完全一致
    （典型 2416 字节）—— 差别只在权重 payload 和描述符
  · 这就是"原生流式"的判别标志，不是折叠，也不是程序变了
```

## 3. 折叠路径（Fold）的完整流程

```
┌──────────────────────────────────────────────────────────────────┐
│                     折叠路径（Fold）                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  起点：权重 blob（压缩 payload）                                  │
│  · int8 (M1):    int8 字节 + 每通道 FP16 scale                   │
│  · 块级 (M1/M2): int8 字节 + 每 block FP16 scale                 │
│  · 任何不满足 Q1–Q7 的格式（即使新代芯片也可能命中）              │
│                                                                  │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼  dispatch 之前（在 host 端做）
┌──────────────────────────────────────────────────────────────────┐
│  F1: host 端 dequantize-to-dense                                  │
│  · 把压缩 payload 展开成 dense FP16 张量                          │
│  · int8:    q → s·(q-z)，每元素写回 FP16                         │
│  · 块级:    q → s_block·q                                         │
│  · 这一步**完全在 host CPU 上跑**，不占引擎周期                   │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  F2: kernel-memory 上限 = 64 KB (HAL[0x200])                     │
│  · 单层权重不得超过 64 KB，否则被 split-legalize 切碎             │
│  · 切碎 → dispatch 数变多 + 编译时间变长                          │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  F3: OCG 记录的 weights 段是 dense FP16                          │
│  · 不再是压缩字节                                                 │
│  · palette LUT 段可能空（如果原本是 int4 被折叠）                 │
│  · relocation 槽仍然填 4 个系数流基地址                            │
│    但 palette LUT 流可能没数据                                    │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼  每次 dispatch
┌──────────────────────────────────────────────────────────────────┐
│  F4: dense FP16 字节通过普通 DMA 搬到芯片                         │
│  · 字节数 = dense FP16 大小                                       │
│  · **跟没压缩的权重带宽占用一模一样**                              │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  F5: 片上无解压（直通）                                           │
│  · 跳过 S4 那一步                                                 │
│  · 直接送进 MAC 阵列                                              │
└──────────────────────────┬───────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│  F6: FP16 送进 MAC 阵列                                           │
│  · 走和 dense FP16 完全相同的乘加 + 宽累加器                       │
│  · 累加精度无损                                                  │
└──────────────────────────────────────────────────────────────────┘

★ 失败指纹：
  · 程序字节跟"dense FP16 编译"的程序字节几乎一致
  · weights 段是 dense FP16 字节
  · M1 上 int8 的延迟 = FP16 的延迟（1.0×），证明走的是这条路
```

## 4. M1 上的格式 → 路径对照表

| 格式 | 决策路径 | M1 走哪条 | A14+ | A15+ | M5/H17s |
|---|---|:-:|:-:|:-:|:-:|
| **int4 LUT** | Q1✓ Q2✓ Q3✓ Q4✓ Q5✓ Q6✓ Q7✓ | **流式** | 流式 | 流式 | 流式 |
| **结构化稀疏** | Q1✓ Q2✓ Q3✓ Q4✓ Q5✓ Q6✗（不走 palette）| **流式** | 流式 | 流式 | 流式 |
| **int8 仿射** | Q1✓ Q2✓ Q3✓ Q4✓ Q5✓ Q6✓ Q7✗（`0x529=0`）| **折叠** | 流式 | 流式 | 流式 |
| **块级仿射** | Q1✓ Q2✓ Q3✓ Q4✓ Q5✓ Q6✓ Q7✗（`0x520–0x539` 清零）| **折叠** | 折叠 | 流式 | 流式 |

> **关键**：M1 上 int8 / 块级被折叠的根因是**副开关 `0x529` / `0x520–0x539` 清零**，**不是主开关 `0x48f` 关闭**。主开关 M1 已经置位，所以 int4 LUT 和 sparse 能流。

## 5. 两条路径的差异对照表

| 维度 | 流式（Stream） | 折叠（Fold） |
|---|---|---|
| **决策条件** | Q1–Q5 全过 + palette 路径 Q6/Q7 也过 | Q1–Q7 任一失败 |
| **kernel-memory 上限** | 16 MB（HAL `0x210`） | 64 KB（HAL `0x200`） |
| **权重段在 OCG 记录里** | 压缩字节（int8/int4/mask+nonzeros） | dense FP16 字节 |
| **dispatch 前是否在 host 展开** | 否 | **是**（F1 步） |
| **DMA 搬运内容** | 压缩字节 | dense FP16 字节 |
| **片上解压** | 有（S4，硬连线） | 无（直通） |
| **带宽节省** | 有（取决于压缩比） | **无** |
| **延迟收益** | 有（M1 int4 LUT 2.37×、sparse 1.55–1.64×） | 无（M1 int8 = 1.0× fp16） |
| **磁盘节省** | 有 | 有（但通常不省带宽） |
| **权重可变（训练中）** | 不允许（Q5 拒绝） | 允许 |
| **dilation / tile 重叠** | 不允许（Q4 拒绝） | 允许 |
| **非 unit stride** | 不允许（Q3 拒绝） | 允许 |
| **M1 上哪些格式走这条** | int4 LUT、结构化稀疏 | int8 仿射、块级仿射 |
| **程序描述符位** | stream enable + palette enable 都置位 | 通常都清零 |

## 6. 三条最值得记住的判断

1. **决策树是串联的**——Q1 到 Q7 每一道都必须过，**任一失败就走折叠**。这是"硬条件集合"，不是"加权打分"。

2. **副开关 `0x529` 是 M1 上 int8/块级被折叠的唯一原因**——主开关 `0x48f` 在 M1 已经置位，但 `0x520`–`0x539` 整簇清零意味着 palette kernel-stride enable 关闭，于是 Q7 失败。这就是为什么 M1 上 int8 看起来"主开关都开了还是流不了"。

3. **折叠不是失败，是降级**——折叠路径仍然能跑、能产生正确结果，只是**带宽一字节没省**。M1 上的 int8 fold 仍然给出了正确输出，只是延迟 = fp16。如果只关心磁盘体积（不关心带宽），折叠是可接受的；如果关心推理延迟，必须挑流式格式。

## 6. 一张图浓缩两条路径

```
                    权重 blob（压缩 payload）
                            │
                            ▼
                    ┌───────────────┐
                    │  决策树 Q1-Q7  │
                    └───────┬───────┘
                            │
              ┌─────────────┴─────────────┐
              │                            │
        Q1-Q7 全过                  任一失败
              │                            │

    │ S1  16 MB cap     │        │ F1  host 端展开    │
    │ S2  压缩形态 OCG  │        │ F2  64 KB cap      │
    │ S3  压缩字节 DMA  │        │ F3  dense FP16 OCG │
    │ S4  片上硬连线解压│        │ F4  dense FP16 DMA │
    │ S5  FP16 进 MAC   │        │ F5  直通           │
    │                   │        │ F6  FP16 进 MAC    │
    │ ★ 省带宽          │        │ ★ 不省带宽         │
    └─────────┬─────────┘        └─────────┬─────────┘
              │                            │
              └────────────┬───────────────┘
                           │
                           ▼

                  同一组宽累加器
                  同一份精度保证
```
