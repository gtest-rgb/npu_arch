## 0. 为什么单独拎出来

第25章按主题（格式 / 开关 / 打包 / 边界）组织，但权重本身是**按时间顺序**走过这些主题的。把它拆成"一生"会看到：

> **权重的一生是单向的、不可逆的**——每一步都有信息丢失或物理形态转变，最终在 MAC 阵列门口还原成 FP16。理解这条流水线就理解了"压缩到底省在哪里、丢在哪里、卡在哪里"。

## 2. 每个阶段的关键细节

### 阶段 0：训练好的 FP32 权重

- **物理位置**：host 端 DRAM（来自 ckpt 文件）
- **大小**：4 字节 × `out_ch × in_ch × kh × kw`
- **此刻状态**：纯数学对象，没碰过 ANE 任何 codec

### 阶段 1：Core ML Tools 做量化 / 调色板化 / 稀疏化

四种 codec 各自的"产出物"：

| codec | 产出物 |
|---|---|
| int8 仿射 | `q (int8)` + `s (fp16)` + `z (fp16)`，M1 强制 `z=0` |
| int4 LUT | `indices (4-bit)` + `codebook (16×fp16)` |
| 结构化稀疏 | `mask (1-bit/elem)` + `nonzeros (fp16 packed)` |
| 块级仿射 | `q (int8)` + `s_per_block (fp16)` |

### 阶段 2：format enum 是"身份证号"

enum 范围 1–31，决定 5 张并行查找表的输出：

| 查找函数 | 返回 |
|---|---|
| `ZinKernelFormatGetBitDepth(f)` | 每元素几位 |
| `ZinKernelFormatGetPaletteFormat(f)` | 调色板子格式（仅 enum 7~27） |
| `ZinKernelFormatGetUnderlyingType(f)` | int8 / uint8 / e4m3 / fp16 |
| `ZinKernelGetPaletteLUTSize(f, n)` | 码本字节 × 个数 |
| `ZinKernelFormatGetTypeno(f)` | 描述符里的 typeno |

两个特殊区间：
- **调色板**：enum 7~27（`f - 7 < 0x15`）
- **Unity**（恒等/只缩放）：enum 28~30（`f - 0x1c < 3`）

### 阶段 3：scale/bias 折叠（关键优化，最容易被忽略）

这是论文最强调的"零开销"特性。**编译期**就把 scale 和 bias 折进权重系数，运行时引擎只在 gain-offset 阶段套用：

```
y = s_out · Σ(w'_k · x_k) + b_out
```

含义：
- 反量化 scale 不是运行时乘——是**预先 bake 进权重**
- 对称量化卷积在 M1 上**不为反量化付一分钱运行时开销**
- M1 强制 `z=0` 正是为了让这个折叠成立（非对称的 `q - z` 不好折）

### 阶段 4：组装权重 blob（六段）

```
t%u       → 元信息
lut:      → 激活函数 LUT（43 项 FP16：2 input-clamp + 1 mode + 1 scale
            + 33 knot + 4 尾部外推 + 1 打包 mode + 1 余）
pal:      → 调色板 LUT（4-bit 32B / 8-bit 512B）
ocgs:     → OCG 记录串（见阶段 6）★
scale:    → 每输出通道 scale（已折进权重，但仍然单独存一份）
bias:     → 每输出通道 bias
weights:  → 打包好的 payload（q / indices / mask+nonzeros）
```

**关键约束**：激活 LUT 和调色板 LUT **平级**，共享同一块片上内存预算——一方占得多，另一方就得少占。

### 阶段 5：流式 vs 折叠决策

这是"一生"中最关键的岔路。同一个权重 blob，走哪条路决定**到底省不省带宽**。

**两道开关**：

| 开关 | 偏移 | 含义 |
|---|---|---|
| 主开关（kernel-streaming master） | `0x48f` | A13+ 置 1 |
| 副开关（palette kernel-stride enable） | `0x529` | 专门给调色板用 |

**额外约束**（主开关之外）：
- 所有轴 stride = 1
- 无 dilation、无 tile 重叠
- 权重**不可变**

**M1 真值表**：

| 格式 | `0x48f` | `0x529` | 路径 |
|---|:-:|:-:|---|
| int4 LUT | ✓ | ✓ | **流式** |
| 结构化稀疏 | ✓ | — | **流式**（独立操作数） |
| int8 仿射 | ✓ | ✗ | **折叠** |
| 块级仿射 | ✓ | ✗ | **折叠** |

**kernel-memory 上限双轨制**：

| 路径 | 上限 | HAL offset |
|---|---|---|
| 流式 | **16 MB** | `0x210` |
| 折叠 / dense | **64 KB** | `0x200` |

> 流式不只是省带宽，还**避免权重被切碎**——拆层会增加 dispatch 数和编译时间。

### 阶段 6：OCG 打包（per-output-channel-group record）

权重段切成一串**固定 14 字**的块：

| 字号 | 字段 | 初始值 |
|---|---|---|
| `[0]` | 该 OCG 在段内的字节偏移 | 编译期算出 |
| `[1]` | 该 OCG 的通道数 | 编译期算出 |
| `[2]`–`[0xb]` | **10 个地址重定位槽** | `-1` |
| `[0xc]`, `[0xd]` | 保留标志位 | `0` |

**M1 特别之处**：kernel 系数**每核心一份副本**（4 核 M1 = 4 份独立 OCG 向量）。
**A14+**：改用共享 kernel 内存布局。

### 阶段 7：Loader 现场填地址（relocation）

10 个 `-1` 槽不是占位符，是**等 loader 在程序加载时填实际地址**。最关键的 4 个系数流基地址对应芯片里 4 个 kernel DMA 子通道：

| 描述符槽 | Enable 寄存器 | Base-offset 寄存器 | Relocation 寄存器 |
|---|---|---|---|
| bias | `0x5548` | `0x554c` | `0x1554` |
| post-scale | `0x5558` | `0x555c` | `0x1558` |
| palette LUT | `0x5568` | `0x556c` | `0x155c` |
| activation LUT | `0x5578` | `0x557c` | `0x1560` |

> **M1 没有 zero-point 流**——因为对称量化 `z=0`。

**这就是为什么"自编译二进制加载不了"**：loader 必须配合守护进程签名路径才能正确填地址（参考第8章管控点④）。

### 阶段 8：DMA 搬运

每次 dispatch：
- 流式路径 → 压缩字节通过 kernel DMA 子通道搬到芯片
- 折叠路径 → dense FP16 字节通过普通 DMA 搬到芯片

> **每次推理都重搬**——权重默认不在片上常驻（除非用 resident buffer + cross-dispatch residency，那是另一条路径）。

### 阶段 9：片上硬连线解压

流式路径在芯片里硬连线做：

| codec | 片上动作 |
|---|---|
| int8 | `q → s·q`（M1 对称）或 `s·(q-z)` |
| int4 LUT | `k → codebook[k]`（纯查表，无算术） |
| sparse | `mask + nonzeros → scatter 回 dense tile` |
| blockwise | `q → s_block·q` |

**折叠路径这步在 host 端做完了**，芯片直接吃 dense FP16。

### 阶段 10：送进 MAC 阵列

不管存的是什么格式，**解压后永远是 FP16**——然后走和 dense FP16 完全相同的乘加 + 宽累加器。

这就是论文反复强调的安全性来源：
- **累加精度无损**（宽累加器保住了）
- **唯一损失是权重量化误差本身**
- 对 cancellation-heavy 的层（残差里大量正负抵消），任何低精度形式都不安全——这跟第3章数值讨论一致

## 3. 三个特别值得记住的判断

1. **"权重的一生"是单向不可逆的**——训练 → codec 选型 → 编译 → 打包 → 加载 → 搬运 → 解压 → MAC。每一步都有不可逆的"压缩"——scale/bias 在编译期折进系数后，原始 FP32 信息就丢了。所以阶段 1 的 codec 选择是**全链路最关键的决定**。

2. **阶段 5（流式 vs 折叠决策）是带宽收益的命门**——同一个权重 blob 走哪条路决定它**到底省不省带宽**。M1 上的 int8 在阶段 5 被判"折叠"，于是阶段 8 搬的是 dense FP16 字节，**前面的压缩等于白做**（只省了磁盘）。

3. **阶段 7（Loader 填地址）是"加载时签名校验"的物理基础**——4 个系数 DMA 子通道的基地址必须在程序加载时由 loader 填进去，loader 必须配合守护进程签名路径。这就是为什么"自编译二进制加载不了"——不是引擎不认，是 loader 路径根本走不通。

## 5. 与其他章节的勾连

- **→ 第7章（Weights and compression）**：第7章是用户视角的"压缩能省什么"，本章是同主题的"权重怎么走完这条路"
- **→ 第3章（Numerics）**：阶段 9–10 的"还原成 FP16 + 宽累加器"是第3章数值安全性的物理基础
- **→ 第8章（Entitlement boundary）**：阶段 7 的 loader 填地址是第8章管控点④（load-time 双校验）的执行层
- **→ 第24章（HAL and capability gates）**：阶段 5 的两道开关（`0x48f` 主 + `0x529` 副）是第24章 HAL 能力字节的具体实例
- **→ 第20章（Winograd）**：sparse / unity 权重在阶段 5 之前就被排除在 Winograd 路径外
