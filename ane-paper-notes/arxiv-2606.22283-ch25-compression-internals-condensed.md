## 1. 四种压缩格式（核心表）

| 格式 | 反量化公式 | 在线形态 | 哪代芯片原生流式 | 编译器开关 |
|---|---|---|---|---|
| **int8 仿射**（per-tensor / per-channel） | `w = s·(q − z)`（M1 强制 `z = 0`） | int8 字节流 + 每通道 FP16 scale | A14+ | M1 折叠到 dense |
| **int4 查找表**（LUT / 调色板） | `w = LUT[g/v][k][c mod v]` | 4-bit 索引流 + 16 项 FP16 码本 | M1/H13+ | 流式主开关 + palette 副开关 |
| **结构化稀疏** | mask 把非零值 scatter 回原位 | 1-bit mask + 压缩 FP16 非零值 | M1/H13+ | 流式主开关，作为独立操作数 |
| **块级仿射**（blockwise） | `w = s_block·q`，每 block 一组 scale | int8 字节流 + 每 block FP16 scale | A15+ | M1/M2 折叠到 dense |

### 1.1 四种格式的关键细节

**int8 仿射**
- M1 上强制对称：`z = 0`，公式退化成 `w = s·q`
- 非对称设置在 M1 上**直接硬断言失败**（编译期，非运行时）
- 每通道 scale 和 bias 在编译期折进权重系数，运行时无反量化开销

**int4 LUT / 调色板**
- 类型表里**没有 4-bit 标量类型**——4 位只能是索引，不可能是数值
- 索引位宽支持 2/4/6/8 bit，但 M1 实际只跑 **4-bit（16 项）** 和 **8-bit（256 项）**
- 调色板向量大小 `v ∈ {1, 2, 4, 5}`（判断：`2^v & 0x36` 命中即合法）
- 4-bit 码本 32 字节，8-bit 码本 512 字节

**结构化稀疏**
- `mask`: `ceil(n/8)` 字节，每元素 1 bit，LSB 优先
- `nonzeros`: `2·popcount(mask)` 字节，FP16 非零值按扫描顺序紧凑打包
- M1 实测：6× 卷积栈、63% 零密度 → 0.43× dense 字节、**1.55–1.64× 加速**

**块级仿射**
- 跟 int8 几乎一样，但 scale 粒度更细（每 block 而非每通道），量化误差更低
- M1/M2 无原生流式编码 → dequantize-to-dense

### 1.2 一份权重 blob 的内部结构

每个编译好的权重 blob 开头有一个 `format enum`（1–31 的"身份证号"），以它为 key 查 5 张并行表：

| 查找函数 | 返回 |
|---|---|
| `ZinKernelFormatGetBitDepth(f)` | 每元素占几位 |
| `ZinKernelFormatGetPaletteFormat(f)` | 调色板子格式（仅对 enum 7~27 有效） |
| `ZinKernelFormatGetUnderlyingType(f)` | 底层类型（int8 / uint8 / e4m3 / FP16） |
| `ZinKernelGetPaletteLUTSize(f, n)` | 码本字节 × 码本个数 |
| `ZinKernelFormatGetTypeno(f)` | 写进描述符的类型编号 |

两个关键区间：
- **调色板格式**：enum 7~27（`f - 7 < 0x15`）
- **Unity 格式**（恒等/只缩放，跳过 MAC）：enum 28~30（`f - 0x1c < 3`）

一份权重由六段组成：

```
t%u       → 元信息
lut:      → 激活函数 LUT
pal:      → 调色板 LUT
ocgs:     → 输出通道分组表（OCG）★
scale:    → 每输出通道 scale
bias:     → 每输出通道 bias
weights:  → 打包好的权重或索引数据
```

> **注意**：激活 LUT 和调色板 LUT **平级**，共享同一块片上内存预算。

## 2. 流式 vs 折叠：两道开关的决策

### 2.1 两条路

- **Stream（流式）**：压缩字节直接 DMA 到芯片，**片上硬连线**现场解压 → 省带宽
- **Fold（折叠）**：dispatch 前在 host 端展开成 dense FP16 → 不省带宽

### 2.2 两道协作开关

**主开关：`0x48f`（kernel-streaming master）**
- 从 A13 起置 1（M1 已置位）
- 必须同时满足：
  1. `hal[0x48f] = 1`
  2. 格式是 primary 或可由 DMA 转 FP16
  3. 所有轴 stride = 1
  4. 无 dilation、无 tile 重叠
  5. 权重**不可变**（训练参数不能流式）

**副开关：`0x520`–`0x539`（palette 路径）**
- 专给调色板用
- 关键位 `0x529`（per-format kernel-stride-enable）= 1 才真正打开 palette-enable + stream

### 2.3 M1 上的真值表

| 格式 | `0x48f` | `0x529` | M1 结果 |
|---|:-:|:-:|---|
| int4 LUT | ✓ | ✓ | **流式** |
| 结构化稀疏 | ✓ | — | **流式**（走主开关，作为独立操作数） |
| int8 仿射 | ✓ | ✗（`0x520`–`0x539` 清零） | **折叠** |
| 块级仿射 | ✓ | ✗ | **折叠** |

### 2.4 带宽预算的"放松"

| 路径 | kernel-memory 上限 | HAL offset |
|---|---|---|
| 流式权重 | **16 MB** | `0x210` |
| 折叠 / dense 权重 | **64 KB** | `0x200` |

→ 流式路径每层能塞下更多权重，**避免被切碎**（拆层会增加 dispatch 数和编译时间）。

## 3. M1 的能力边界与陷阱

### 3.1 版本矩阵（M1 vs 后续芯片）

| 能力 | M1 version-5 描述符 | 后续描述符 |
|---|---|---|
| sparse-binary 索引打包 | **assert 不支持** | 描述符 `0x424` 一个 packed flag |
| 多码本调色板 | **assert 不支持** | 已实现 |
| 非对称量化 | **assert 不支持** | 已实现 |
| zero-skip 检测 | 已实现 | 已实现 |
| 调色板使能 + 位宽 | 4/8 bit | 4/8 + 3/6 bit |

**M1 合法 kernel-format 空间**（与 version-5 描述符的交集）：
- dense FP16
- 4-bit / 8-bit 调色板（含单码本向量形式）
- 对称 int8（折叠到 FP16）
- Unity 格式

**M1 完全没有**：sparse-binary 索引打包、多码本调色板、非对称量化、FP8、3-bit/6-bit 调色板宽度。

> **关键区分**：这些是**缺 encoder**（编译期就没有），**不是运行时拒绝**。

### 3.2 陷阱 1：两套稀疏机制不要搞混

引擎里有两套**独立**的稀疏：

| 机制 | 怎么工作 | 收益 | 适用条件 |
|---|---|---|---|
| **① zero-skip**（计算时跳零） | 表描述符 `detect-zeros` 位打开 → MAC 遇到权重 0 跳过 | 带宽受限层**仅 ~1%**（瓶颈在字节流不在 MAC） | 任何含 0 的 kernel 都吃，包括 fold 成 dense 的 |
| **② sparse binary + 片上解压** | mask + nonzeros DMA 流过来，片上电路 scatter 回 dense tile | M1 实测 **1.55–1.64×** | 独立操作数走主开关 `0x48f` |

### 3.3 陷阱 2：那个看似矛盾的 sparse assert

version-five 表描述符**断言 sparse-binary 模式不支持**，但实际 sparse 又能流。为什么？

- 那个 assert 管的是**另一条**路径——调色板索引平面内部的 sparse 打包（只有走 `vector-palette` 标志才能到达）
- 能流式的 sparse 走的是 §25.4 那条**独立的 mask+values 操作数路径**，**不经过**那个表描述符位

### 3.4 陷阱 3：FP8——M1 看得到用不了

编译器里有完整 FP8 通路，但 M1 不能跑：

| 形式 | 元素代码 | 角色 |
|---|---|---|
| **E4M3** | `0xc` | gated 硬件权重 + 激活格式 |
| **E5M2** | `0xd` | 转换格式 |

E4M3 是被门控的那个：
- 内核格式合法性**承认**它是原生权重元素类型
- 但写它元素代码的 **encoder，A17 起才存在**
- 运行时能力字节 `0x52d` **只在 A18 上置位**
- M1 上格式寄存器只有 2 位宽，**没有 E4M3 的码点** → **编译期 assert**（不是运行时拒绝）

> 累加器永远是宽 FP16-class——FP8 只是输入/存储宽度，喂给同一组 MAC。吞吐走跟 int8 同样的"双乘"路径。

### 3.5 陷阱 4：Winograd 排除 sparse 和 unity

Winograd 加速小卷积（典型 3×3）时，**资格要求权重非 unity、非 sparse**——sparse 或 unity 格式直接排除在 Winograd 路径外，走自己的数据通路。变换矩阵 `G` / `B⊤` / `A⊤` 常驻阵列，不出现在任何可发货权重 blob 里。

## 4. 三个值得记住的判断

1. **M1 上只有 int4 LUT 和结构化稀疏真省带宽**——int8 / 块级仿射被 fold 到 dense FP16，磁盘小但 DRAM 搬运一字节没省。从 A14 起 int8 才流，A15 起块级才流。

2. **"缺 encoder" ≠ "运行时拒绝"**——M1 的 sparse-binary 索引、多码本、非对称量化、FP8、3/6-bit 调色板都是**编译期就没有代码路径**，比运行时拒绝更彻底。改 entitlement / 改 IR 都救不了。

3. **两套稀疏机制是独立的**——zero-skip 是"算得快"（但带宽受限时收益微乎其微），sparse binary + 片上解压才是"搬得少"。前者跟存储格式无关，后者要求独立操作数 + 主开关。
