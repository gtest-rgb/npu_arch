## 1. 支持的压缩类型

引擎认识**且仅认识**四种压缩权重格式——和 Core ML Tools 文档化的"模型体积压缩"用的是同一套。

### 1.1 总览表

| 格式 | 反量化公式 | 在线形态 | 哪代芯片原生流式 | M1 上 |
|---|---|---|---|---|
| **int8 仿射**（per-tensor / per-channel） | `w = s·(q − z)`（M1 强制 `z = 0`） | int8 字节流 + 每通道 FP16 scale | A14+ | **折叠** |
| **int4 查找表**（LUT / 调色板） | `w = LUT[g/v][k][c mod v]` | 4-bit 索引流 + 16 项 FP16 码本 | M1/H13+ | **流式** |
| **结构化稀疏** | mask 把非零值 scatter 回原位 | 1-bit mask + 压缩 FP16 非零值 | M1/H13+ | **流式** |
| **块级仿射**（blockwise affine） | `w = s_block·q`，每 block 一组 scale | int8 字节流 + 每 block FP16 scale | A15+ | **折叠** |

### 1.2 每种格式的关键细节

**int8 仿射（最经典）**
- M1 强制对称：`z = 0`，公式退化成 `w = s·q`
- 非对称设置在 M1 上**编译期硬断言失败**
- 每通道 scale 和 bias 在编译期折进权重系数，运行时无反量化开销

**int4 LUT / 调色板（M1 唯一原生流式的"密集"格式）**
- 类型表里**没有 4-bit 标量类型**——4 位只能是索引，不可能是数值
- 索引位宽支持 2/4/6/8 bit，但 M1 实际只跑 **4-bit（16 项）** 和 **8-bit（256 项）**
- 调色板向量大小 `v ∈ {1, 2, 4, 5}`（判断：`2^v & 0x36` 命中即合法）
- 4-bit 码本 32 字节，8-bit 码本 512 字节

**结构化稀疏**
- `mask`: `ceil(n/8)` 字节，每元素 1 bit，LSB 优先
- `nonzeros`: `2·popcount(mask)` 字节，FP16 非零值按扫描顺序紧凑打包
- M1 实测：6× 卷积栈、63% 零密度 → 0.43× dense 字节、**1.55–1.64× 加速**

**块级仿射**
- 跟 int8 几乎一样，但 scale 粒度更细（每 block 而非每通道），量化误差更低
- M1/M2 无原生流式编码 → dequantize-to-dense

### 1.3 一份权重 blob 的内部结构

每个编译好的权重 blob 开头有一个 `format enum`（1–31 的"身份证号"），以它为 key 查 5 张并行表：

| 查找函数 | 返回 |
|---|---|
| `ZinKernelFormatGetBitDepth(f)` | 每元素占几位 |
| `ZinKernelFormatGetPaletteFormat(f)` | 调色板子格式（仅对 enum 7~27 有效） |
| `ZinKernelFormatGetUnderlyingType(f)` | 底层类型（int8 / uint8 / e4m3 / FP16） |
| `ZinKernelGetPaletteLUTSize(f, n)` | 码本字节 × 码本个数 |
| `ZinKernelFormatGetTypeno(f)` | 写进描述符的类型编号 |

两个关键区间：
- **调色板格式**：enum 7~27（`f - 7 < 0x15`）
- **Unity 格式**（恒等/只缩放，跳过 MAC 阵列）：enum 28~30（`f - 0x1c < 3`）

一份权重由固定六段组成：

```
t%u       → 元信息
lut:      → 激活函数 LUT（与 pal 共享片上预算）★
pal:      → 调色板 LUT（4-bit/8-bit 码本）
ocgs:     → 输出通道分组表（OCG）★ 核心数据结构
scale:    → 每输出通道 scale
bias:     → 每输出通道 bias
weights:  → 打包好的权重 / 索引 / mask+nonzeros
```

> **注意**：激活 LUT 和调色板 LUT **平级**，共享同一块片上内存预算——一方占得多，另一方就得少占。

## 2. 压缩权重的一生

权重从训练好到送进 MAC 阵列，要走过 10 个阶段。**单向不可逆**——scale/bias 在编译期折进系数后，原始 FP32 信息就丢了。

│  · 形状 [out_ch, in_ch, kh, kw]，每个元素 4 字节                  │

│  · 选 4 种格式之一（int8 / int4 LUT / sparse / blockwise）         │
│  · 算 scale / zero-point / 码本 / mask                            │
└──────────────────────────┬──────────────────────────────────────┘
                           │

### 2.1 一生中最关键的三个岔路

| 岔路 | 阶段 | 决定什么 | 不可逆性 |
|---|:-:|---|---|
| **codec 选择** | 1 | 反量化公式、码本 / mask / scale 的形态 | 一旦量化，原始 FP32 信息丢失 |
| **scale/bias 折叠** | 3 | 编译期 bake 进系数，运行时零开销 | 折叠后无法分离 |
| **流式 vs 折叠** | 5 | 走哪条物理路径、省不省带宽 | 程序描述符位已写死，运行时不可切换 |

### 2.2 三条最值得记住的判断

1. **"权重的一生"是单向不可逆的**——scale/bias 一旦折进系数，原始 FP32 信息就丢了。阶段 1 的 codec 选择是全链路最关键的决定。

2. **阶段 5（流式 vs 折叠决策）是带宽收益的命门**——同一个权重 blob 走哪条路决定它**到底省不省带宽**。M1 上的 int8 在阶段 5 被判"折叠"，于是阶段 8 搬的是 dense FP16 字节，**前面的压缩等于白做**（只省了磁盘）。

3. **阶段 7（Loader 填地址）是"加载时签名校验"的物理基础**——4 个系数 DMA 子通道的基地址必须在程序加载时由 loader 填进去，loader 必须配合守护进程签名路径。这就是为什么"自编译二进制加载不了"——不是引擎不认，是 loader 路径根本走不通。

## 4. 与其他章节的勾连

- **→ 第7章（Weights and compression）**：第7章是用户视角的"地图"（stream vs fold + 选型），本章是同主题的内部剖面
- **→ 第3章（Numerics）**：阶段 9–10 的"还原成 FP16 + 宽累加器"是第3章数值安全性的物理基础
