## Context

当前 `npu-performance-optimization` spec 第4节描述了 Vector 和 Cube 的级联优化，但存在概念混淆：

**问题根源**：将 FixPipe 的随路处理能力与独立 Vector 算子混为一谈

```
当前错误理解:
┌─────────────────────────────────────────────────────────────────┐
│ Cube → Vector 级联场景:                                         │
│   - 激活函数 (GELU, ReLU)                                       │
│   - 归一化                                                      │
│   - 反量化                                                      │
│ 通路: FixPipe (L1 → UB)                                         │
└─────────────────────────────────────────────────────────────────┘

正确理解:
┌─────────────────────────────────────────────────────────────────┐
│ FixPipe 随路处理 (Cube 内部能力):                                │
│   - 量化、反量化                                                 │
│   - 激活函数 (ReLU 等)                                          │
│   - 其他可配置的后处理                                           │
│                                                                 │
│ Cube → Vector 级联 (需要独立 Vector 阶段):                       │
│   - Softmax                                                     │
│   - LayerNorm                                                   │
│   - Add, Mul 等一般计算                                          │
│ 通路: FixPipe (L1 → UB)                                         │
└─────────────────────────────────────────────────────────────────┘
```

## Goals / Non-Goals

**Goals:**
- 修正第4节的概念描述，区分 FixPipe 随路处理与独立 Vector 算子
- 更新典型算子示例，确保示例准确反映实际场景
- 保持文档结构不变，仅修正描述内容

**Non-Goals:**
- 不改变文档的整体结构和章节划分
- 不添加新的级联模式
- 不涉及代码实现变更

## Decisions

### 决策1: 保留三种级联模式分类

**理由**: 模式分类本身是正确的，问题在于场景描述不准确

- Vector → Cube: 前置 Vector 算子 + Cube（保留）
- Cube → Vector: Cube + 后置 Vector 算子（修正场景描述）
- 双向级联: Vector → Cube → Vector（修正场景描述）

### 决策2: 添加 FixPipe 随路处理说明

**位置**: 在 4.1 级联场景概览后添加新小节

**内容**: 说明 FixPipe 作为 Cube 随路处理的能力边界

```
┌─────────────────────────────────────────────────────────────────┐
│ FixPipe 随路处理 vs 独立 Vector 算子                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│ FixPipe 可随路处理 (无需独立 Vector 阶段):                       │
│   ✓ 量化 / 反量化                                               │
│   ✓ ReLU, LeakyReLU 等简单激活                                  │
│   ✓ Bias 加法                                                   │
│   ✓ 格式转换                                                    │
│                                                                 │
│ 需要独立 Vector 算子 (需要 Cube→Vector 级联):                    │
│   ✓ Softmax                                                     │
│   ✓ LayerNorm, BatchNorm                                        │
│   ✓ GELU, SiLU 等复杂激活                                       │
│   ✓ 一般向量计算 (Add, Mul, Div 等)                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 决策3: 修正典型算子示例

**4.6 示例2 修改**:
- 原: MatMul → GELU (描述为需要 Cube→Vector 级联)
- 改: MatMul → GELU 可通过 FixPipe 随路处理（如果硬件支持）或 Cube→Vector 级联

**4.6 示例3 修改**:
- 原: Quantize → MatMul → Dequantize (描述为双向级联)
- 改: MatMul + FixPipe 随路量化/反量化（无需独立 Vector 阶段）

**添加新示例**: MatMul → Softmax (真正的 Cube→Vector 级联场景)

## Risks / Trade-offs

**风险1**: 读者可能对 "FixPipe 随路处理" 和 "Cube→Vector 级联" 的边界仍感困惑

→ **缓解**: 使用明确的对比表格，列出具体操作属于哪一类

**风险2**: 修正后可能影响已有读者的理解

→ **缓解**: 这是文档修正，早期发现并修正比传播错误理解更好
