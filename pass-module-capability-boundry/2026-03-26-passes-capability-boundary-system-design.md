# Passes模块能力边界判断体系设计文档

> 版本：1.0
> 日期：2026-03-26
> 作者：Passes模块负责人

## 1. 背景与目标

### 1.1 问题背景

Passes模块是PyPTO编译器的核心组件，包含40+个Pass，分布在Tensor Graph、Tile Graph、Block Graph三个IR层级。作为模块负责人，经常面临以下场景：

- **团队内部评审**：讨论设计方案时，需要快速判断"这个场景Passes能否正确处理"
- **对外技术支持**：用户/开发者提问时，需要快速回答"是否支持、有何限制"

**当前痛点：**
- 回答问题依赖经验+查代码+问人，效率低且易遗漏
- 缺乏系统性的能力边界知识沉淀
- 40+个Pass的信息分散，难以快速综合判断

### 1.2 目标

建立一套**Passes模块能力边界判断体系**，实现：

1. **分钟级响应**：在技术评审/支持场景下，快速给出能力边界判断
2. **全面性**：覆盖功能正确性和边界约束两个维度
3. **有据可查**：每个判断都能指向具体的文档/代码依据
4. **可展示**：输出可以直接展示给提问者

### 1.3 范围

**在范围内：**
- Passes模块整体能力边界定义
- 单个Pass的能力边界文档化
- 场景- Pass映射索引
- AI辅助判断工具

**不在范围内：**
- 自动代码检测（后续扩展）
- 性能分析（本设计聚焦正确性和约束）

## 6. AI辅助判断Skill

### 6.1 Skill定义

**名称：** `pypto-pass-capability-checker`

**描述：** 用于判断给定场景在Passes模块中的能力边界，输出可行性判断、约束条件和风险提示。

### 6.2 工作流程

```
用户输入场景描述
       ↓
┌──────────────────┐
│ 1. 场景理解阶段   │
│   - 提取关键特征  │
│   - 识别模糊点    │
└──────────────────┘
       ↓
┌──────────────────┐
│ 2. 澄清问答阶段   │ ←── 可选，如果场景描述足够清晰可跳过
│   - 追问关键信息  │
│   - 确认理解      │
└──────────────────┘
       ↓
┌──────────────────┐
│ 3. 知识检索阶段   │
│   - 标签匹配      │
│   - Pass关联      │
│   - 约束条件检索  │
└──────────────────┘
       ↓
┌──────────────────┐
│ 4. 综合判断阶段   │
│   - 可行性判断    │
│   - 风险识别      │
│   - 约束检查      │
└──────────────────┘
       ↓
┌──────────────────┐
│ 5. 输出报告      │
│   - 判断结论      │
│   - 引用依据      │
│   - 检查清单      │
└──────────────────┘
```

### 6.3 输出格式

```markdown
## Passes能力边界判断报告

### 场景理解
- 涉及OPCode: MATMUL, ASSEMBLE, VIEW
- IR层级: Tile Graph
- 场景特征: Cube级联, 小搬大

### 判断结论
**整体可行性：✅ 支持 / ⚠️ 有约束 / ❌ 不支持**

### 相关Pass及约束

| Pass | 状态 | 约束条件 | 风险等级 |
|------|------|----------|----------|
| AssignMemoryType | ⚠️ | 小搬大需维度整除 | 中 |
| MergeViewAssemble | ✅ | - | 低 |

### 必须满足的约束
1. **ASSEMBLE offset对齐** - 必须32B对齐，否则强制走DDR
   - 依据：AssignMemoryType §8.3-C001
2. **嵌套深度限制** - VIEW/ASSEMBLE/RESHAPE嵌套≤3层
   - 依据：AssignMemoryType PreCheck

### 潜在风险点
1. **超大Buffer降级** - 如果ASSEMBLE输出超过UB阈值(35%)，会降级为DDR
   - 依据：AssignMemoryType §8.2-R001

### 建议检查项
- [ ] 检查ASSEMBLE的offset是否32B对齐
- [ ] 检查VIEW/ASSEMBLE嵌套深度
- [ ] 检查小搬大维度整除关系

### 参考文档
- AssignMemoryType能力边界：`.agents/skills/.../09_ASSIGN_MEMORY_TYPE.md#8-能力边界`
- 场景索引：`.agents/skills/.../SC001_matmul_cube_cascade.md`
```

### 6.4 澄清问答示例

当场景描述不清晰时，Skill会追问：

```
我理解你描述的场景涉及Cube级联，需要确认几个问题：

1. 这个场景是在哪个IR层级？（Tensor Graph / Tile Graph / Block Graph）
2. ASSEMBLE的输入来源是什么？（单个MATMUL输出 / 多个MATMUL输出）
3. 是否涉及动态Shape？
4. 预期的数据规模大概多大？
```

### 6.5 Skill触发方式

**仅手动调用**，调用方式：
```
/pypto-pass-capability-checker <场景描述>
```

### 7.6 知识库更新机制

**双重保障机制：**

#### 1. Git Hook（即时提醒）

在 `.git/hooks/pre-commit` 中添加检查逻辑：
- 当 `framework/src/passes/` 目录有代码变更时
- 自动提醒：请检查相关Pass文档是否需要更新

#### 2. CI检查（兜底验证）

在CI流程中添加文档同步检查：
- 检查代码变更的Pass是否有对应文档
- 如果代码变更但文档未更新，输出警告
- 不阻止合并，但作为Review提醒

**更新触发条件：**
- Pass核心逻辑变更（RunOnFunction、PreCheck、PostCheck）
- 新增约束条件
- 新增/修改OPCode特判
- 修复边界相关Bug

## 9. 成功指标

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 响应时间 | 分钟级（<5分钟） | 从接收问题到输出判断报告的时间 |
| 判断准确性 | >90% | 与实际代码验证结果的一致性 |
| 覆盖率 | 覆盖80%常见场景 | 典型场景Case覆盖率 |
| 可追溯性 | 100% | 每个判断都有文档/代码引用 |

## 10. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 知识库维护成本高 | 文档滞后于代码 | Git Hook + CI双重保障机制 |
| AI判断可能有幻觉 | 错误的判断结论 | 输出必须包含引用依据，便于验证 |
| 场景标签体系不完整 | 遗漏重要场景 | 持续收集反馈，迭代扩展标签 |
| Pass数量多，工作量大 | 阶段一耗时长 | 优先完成核心Pass，逐步扩展 |
